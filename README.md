# 一、产品定位与商业模式

## 1.1 产品定位

**一句话**：面向医疗机构的B2B健康科普内容交付系统，用AI规模化生产符合医院审核标准的定制化健康文章，通过"人机协同工作流"实现高通过率交付。

**核心价值主张**：
- **对医院客户**：将内容生产外包从"找写手团队"升级为"AI工业化交付"，成本更低、产能更稳定、质量可标准化
- **对我们**：用技术替代人力写作，将毛利率从传统代运营的20-30%提升到50-70%

## 1.2 商业模式

```
医院/医疗机构（甲方）
    │
    │  固定合同：¥X万 = Y天内交付Z篇
    │  · 科室/疾病方向由甲方指定
    │  · 质量标准 = 甲方审核通过
    │
    ▼
我方团队（乙方）
    │
    ├── 项目/商务：签约、客户对接、需求沟通
    ├── AI系统：批量生成文章
    ├── 内部审核：一审把关（专业度+可读性+合规）
    └── → 交付甲方 → 甲二审 → 通过即发布至甲方百家号
```

**跟"自己做号赚广告费"的本质区别**：

| 维度 | B2C自媒体模式 | B2B交付模式（实际） |
|------|-------------|-------------------|
| 收入来源 | 百家号广告分成CPM | 固定合同金额 |
| 收入确定性 | 不稳定，依赖爆款 | 100%确定，签约即有 |
| 质量标准 | 平台审核+用户用脚投票 | 甲方客户满意=通过 |
| 核心指标 | 阅读量、粉丝增长、完读率 | **交付效率**：通过率×产速 |
| 核心风险 | 平台算法变化 | 客户流失/合同续约 |
| 规模化路径 | 铺量、多账号矩阵 | 拓客、提升产能、缩短交付周期 |

## 2.1 审核链路详解

这是整个服务模式的核心——**两轮审核定义了产品质量标准**：

| 轮次 | 审核方 | 关注点 | 不过的处理 |
|------|--------|--------|------------|
| **一审（内部）** | 我方审核团队 | 文章结构完整性、故事逻辑、标题合规、基本医学常识无误 | 打回修改（AI重生成或人工修改） |
| **二审（甲方）** | 医院客户 | 医学准确性（核心）、是否符合品牌调性、是否符合科室定位 | 驳回修改后重新交付 |

**为什么是两轮？**
- 一轮挡掉明显的低质品，减少甲方审核负担 → 甲方体验好
- 如果AI直接交付甲方，大量可避免的驳回会消耗客户耐心和信任

## 3.2 自回归生成机制（核心技术创新点）

```
❌ 一次性生成全文:
   Prompt ──→ 2500字全文
   问题: LLM会在后半段"忘记"前面的人物设定和情节，故事断裂

✅ 自回归逐段生成:
   扩展Ch1: prompt + Ch1大纲 ──→ Ch1正文
   扩展Ch2: prompt + Ch1正文(已生成) + Ch2大纲 + Ch3大纲 + Ch4大纲 ──→ Ch2正文
   扩展Ch3: prompt + Ch1正文 + Ch2正文 + Ch3大纲 + Ch4大纲 ──→ Ch3正文
   扩展Ch4: prompt + Ch1正文 + Ch2正文 + Ch3正文 + Ch4大纲 ──→ Ch4正文

   关键: 每生成一章，都把前面所有已生成的正文注入到当前prompt中
         模型永远在"看到前文"的状态下写下一段
```

**代价与应对**：
- 延迟×4（4次API调用）→ 通过60并发线程池对冲，总吞吐量可控
- 首段出错传播 → 大纲质量是第一道防线（后来优化中加了大纲预审）

# 四、质量优化：30% → 80-90% 通过率提升

## 4.1 问题诊断阶段

**现象**：初期交付给甲方二审通过率仅≈30%

**首先做的事**：不是急着改代码，而是对100篇被打回的文章做归因分类。

```
驳回原因分布（推测框架，面试时按实际情况填充）：
┌────────────────────────────────────┐
│ 类别A: 医学内容不准确/表述不严谨  ~35%│  ← 最大因子
│ 类别B: 故事逻辑不通/情节不合理   ~25%│
│ 类别C: 标题不符合医院调性        ~20%│
│ 类别D: 内容与指定科室方向无关    ~12%│
│ 类别E: 格式/排版问题             ~8% │
└────────────────────────────────────┘
```

## 4.2 针对性优化动作

**针对A类（医学不准确）—— 最核心的优化**：
- 在prompt中强化"医学表述必须严谨准确"的约束
- 建立疾病→常见误区→正确表述的对照库（与医院专家沟通确认）
- 大纲阶段增加医学逻辑校验（症状→诊断→科普的因果链是否闭环）
- 一审团队配备医学背景审核员

**针对B类（故事逻辑不合理）**：
- 大纲评审机制：生成大纲后，用更便宜的模型（如ds-v3）对大纲做一次快速逻辑审查，低分大纲直接重生成，避免浪费后续4次API调用
- 优化人物设定与健康误区之间的因果关系（职业→行为→疾病必须强逻辑关联）

**针对C类（标题不符合调性）**：
- 与甲方沟通，明确标题规范的具体边界（什么能说、什么不能说）
- 建立标题违禁词/高风险表述的正则过滤规则
- prompt中强化甲方品牌调性约束

**针对D类（内容偏离科室方向）**：
- 话题生成阶段增加"科室标签"字段，指定每篇文章的目标科室
- 大纲prompt中注入科室上下文，确保内容聚焦

## 4.3 审核流程优化

```
优化前:
  AI生成 → 内部一审 → 交付甲方二审（通过率30%）
  问题: 一审标准与甲方标准不对齐，大量不合格品流入二审

优化后:
  AI生成 → 内部一审（对标甲方审核标准）
         → 不通过: 分类驳回原因 → 针对性重生成 → 再审
         → 通过: 交付甲方二审（通过率80-90%）
  
  关键改变:
  1. 一审标准校准——拿甲方驳回样本反哺一审标准
  2. 建立驳回原因分类体系，每种原因有对应的重生成/修改策略
  3. 一审不通过的反馈闭环：驳回原因 → prompt调整 → 重生成质量提升
```

## 4.4 效果与量化

| 指标 | 优化前 | 优化后 | 提升 |
|------|--------|--------|------|
| 甲方二审通过率 | ≈30% | ≈80-90% | +2.7~3倍 |
| 有效产能（相同投入下） | 30篇 | 80-90篇 | +2.7~3倍 |
| 甲方案例平均修改轮次 | 2-3轮 | 0-1轮 | 减少1-2轮 |
| 交付周期 | 基准 | 缩短40-60% | — |

# 五、指标体系（B2B交付视角）

## 5.1 北极星指标

**有效交付产能** = 单次合同周期内甲方通过的文章数

= 生成总量 × (1 - 生成失败率) × 一审通过率 × 二审通过率

选这个而非"生成量"的原因：不过审的文章就是废品，不产生收入。关键是AI生成+审核过滤后的**有效输出**。

## 5.2 过程指标

### 效率指标

| 指标 | 含义 | 当前水位 |
|------|------|----------|
| 单篇端到端耗时 | AI生成+一审修改+二审等待 | - |
| 一审通过率 | 内部筛掉的比例 | - |
| **二审通过率** | **最终交付质量** | **80-90%** |
| 单篇平均修改轮次 | 被驳回后修改几次才通过 | - |
| 日有效产出 | 每天最终通过的文章数 | - |
| 单篇交付成本 | (API成本+审核人力) / 通过数 | - |

### 质量指标

| 指标 | 含义 |
|------|------|
| 驳回原因分布 | 哪个类别最常导致驳回（用于持续优化） |
| 驳回率趋势 | 随合同批次是否持续下降（学习曲线） |
| 甲方满意度 | 主动续约率 / 追加合同金额 |

### 商业指标

| 指标 | 含义 |
|------|------|
| 单合同毛利率 | (合同金额 - 交付成本) / 合同金额 |
| 客户LTV | 单客户累计合同金额 |
| 客户获取成本(CAC) | 商务成本 / 新客数 |
| 产能利用率 | 当前产出 / 最大产能上限 |

# 六、核心决策与Trade-off

| 决策项 | 选择 | 原因 | 代价 |
|--------|------|------|------|
| 生成策略 | 自回归逐段生成 | 长文故事连贯性，甲方严重看这个 | 延迟×4，需要并发对冲 |
| 模型分工 | doubao(大纲) + ds-r1(正文) | doubao中文好便宜，ds-r1推理强 | 跨模型风格可能不一致 |
| 文章结构 | 固定4章结构 | 标准化=质量可控+审核时对照检查 | 模板化风险，特殊话题不适用 |
| 一审前置 | 内部先审再交甲方 | 保护甲方体验，减少无效来回 | 增加内部审核人力成本 |
| JSON解析 | 三层fallback解析 | LLM输出不稳定 | 代码复杂度增加 |
| 并发模型 | 话题ProcessPool / 文章ThreadPool | I/O密集型用线程，混合作业用进程 | 进程池启动开销 |
| 分块策略 | 120篇/chunk | 支持中断恢复+控制内存 | cross-chunk不可复用上下文 |
